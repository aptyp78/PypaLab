# Два Mac Studio, 1 ТБ памяти и призрак EHOSTUNREACH: как мы собирали распределённый inference-кластер

> Черновик. Инженерный детектив о том, как объединить два Mac Studio M3 Ultra (512 ГБ каждый) в один exo-кластер для distributed-inference — и почему почти каждая «очевидная» причина сбоя оказывалась ложной.

## Зачем

Две Mac Studio M3 Ultra, по 512 ГБ unified memory каждая. По отдельности — каждая тянет модели до ~400 ГБ в кванте. Вместе, если объединить память в пул, — **~1 ТБ**, а это уже класс моделей, который физически не влезает в одну машину: DeepSeek-V3/V4, Kimi-K2, Qwen3-Coder-480B в 8-битном кванте.

Инструмент для объединения — [exo](https://github.com/exo-explore/exo): он строит libp2p-меш между узлами и шардит модель по слоям (pipeline-параллелизм). Задача звучала просто: «поднять exo на двух Маках, чтобы они считали как один». На практике это превратилось в неделю отладки, где почти каждая гипотеза о причине сбоя была опровергнута следующим экспериментом.

Это история про дисциплину отладки: как не врать себе, отсекать гипотезы экспериментом, и не чинить то, что не сломано.

## Акт 1. EHOSTUNREACH, которого не должно быть

Первый симптом: узлы поднимаются, видят себя, но **не образуют меш**. `/state` каждого узла показывает один узел — себя. В логах libp2p — `Os { code: 65, kind: HostUnreachable }` при попытке узла `.30` подключиться к `.40`.

`EHOSTUNREACH` — «нет маршрута до хоста». Классика. Проверяем очевидное:

- `ping .40` — идёт.
- `nc -z .40 42040` (libp2p-порт) — **порт открыт**.
- Таблица маршрутов — есть маршрут к подсети.

Сеть работает. Но libp2p внутри exo упорно получает «нет маршрута». Дальше — длинный список опровергнутых гипотез, каждую из которых проверяли **экспериментом, а не рассуждением**:

| Гипотеза | Эксперимент | Вердикт |
|---|---|---|
| Маршрут к `.40` битый | сырой `connect()` из Python к `.40:42040` | ✅ 1 мс — не маршрут |
| Привязка к неверному source-IP | `bind()` к `.30` + connect | ✅ проходит — не source |
| Опции сокета (SO_REUSEADDR, nonblock) | реплика в Python | ✅ проходит — не опции |
| `IP_BOUND_IF` (привязка к интерфейсу) | DYLD-interpose на `connect()` | `IP_BOUND_IF=0` — не привязка |
| Битый бинарь Python | тот же venv-python → `.40` | ✅ проходит — не бинарь |
| namespace/firewall/Tailscale | прямые проверки | все мимо |

Особенно показателен **DYLD-interpose**: подменяем `connect()` своей библиотекой, логируем точные аргументы и `errno`. Видно: сокет ни к чему не привязан (`IP_BOUND_IF=0`), адрес назначения верный, а `connect()` всё равно возвращает 65 — **синхронно**, ещё до отправки SYN. То есть вердикт «нет маршрута» выносит ядро локально, для конкретного процесса.

Ключевой дифференциальный тест: **внутри процесса exo** открываем свежий сокет и коннектимся к `.40` — падает. **В соседнем процессе** (Python, Rust, что угодно) в ту же секунду — проходит 60 из 60. Один и тот же syscall, одна машина, один пользователь — разный результат. Значит дело не в сокете и не в сети, а в **самом процессе**.

### Развязка Акта 1

Что отличает процесс exo? Он стартует внутри tmux-сессии, а tmux-сервер на этой машине был поднят **launchd при загрузке**. Финальный эксперимент:

```
python-connect к .40  напрямую через ssh-сессию  →  OK
тот же connect  внутри launchd-tmux              →  EHOSTUNREACH
```

Вот и причина. macOS назначает процессу сетевой scope (привязку к «основному» интерфейсу/состоянию сети) **в момент старта и наследует от родителя**. launchd поднял tmux-сервер на раннем этапе загрузки, когда сетевое состояние было другим; все потомки этого сервера унаследовали scope, отрезанный от локальной подсети. Процессы из живой ssh-сессии — в чистом scope, ходят свободно.

Лечение оказалось не в коде и не в сети, а в **контексте запуска**: поднять exo в свежем tmux-сервере, порождённом из ssh-сессии (`tmux -L <socket>`), а не в launchd'овском. И меш собрался — `connect` к `.40` пошёл мгновенно.

**Урок:** «процесс A ходит, процесс B нет» при идентичных аргументах — это почти всегда process-level state (scope, политика, унаследованный контекст), а не код. Дифференциальный тест в одном процессе vs в соседнем экономит дни.

## Акт 2. Кластер собрался, но не считает

Меш на двух узлах: `/state` показывает 2 узла, ~1.1 ТБ суммарной памяти, каталог из 85 моделей. Победа? Запускаем инференс — и runner падает на компиляции Metal-кернелов:

```
gather_front.h: use of undeclared identifier 'offset_neg_idx'
utils.h: unknown type name 'bfloat16_t'
```

exo тащит за собой кастомный форк MLX (заточенный под distributed и под DeepSeek-V3.2). Его mlx-core **не компилит свои Metal-кернелы в рантайме**. И именно поэтому кластер за много часов работы ни разу ничего не посчитал — сеть подняли, а движок инференса мёртв.

Проверка гипотезы «железо или форк»: ставим **чистый upstream MLX** и гоняем ту же модель напрямую.

```
upstream MLX, та же модель, то же железо:
Generation: 86 tok/s | Peak memory 32 GB
```

Железо и тулчейн исправны. Виноват **только форк**. Дальше — аккуратная пересборка стека:

- **upstream mlx-core** (рабочие предкомпилированные кернелы) +
- **форкнутый mlx-lm** (нужен exo по API и для поддержки DeepSeek-V3.2) +
- правильная версия `anyio` (runner требует API из 4.x).

Собранный на коленке «гибрид» завёлся: exo обслужил инференс через свой pipeline. **Урок:** «кластер собран» ≠ «кластер считает». Сетевая связность и работающий движок — две независимые проверки, и обе обязательны.

## Акт 3. Кольцо, которого не было

Последняя стена — distributed-инференс на 2 узла требует **цикла в графе топологии** (exo использует кольцевой pipeline). А `placement(min_nodes=2)` упорно отвечает `No cycles found`.

Раскопки в коде показали неочевидное: рёбра топологии exo строит **не из libp2p-соединений**, а из независимой reachability-проверки — каждый узел периодически пингует соседей по HTTP `/node_id` на **захардкоженном порту 52415** и создаёт ребро. Наш кластер жил на порту 52420 → reachability не проходила → рёбер нет → цикла нет.

Плюс два смежных открытия:
- exo по умолчанию **не персистит libp2p-идентичность** (в коде — `return Keypair.generate()` с TODO «вернуть персистентность, когда разберёмся с дубликатами»), поэтому peer-id менялся при каждом рестарте и взаимный bootstrap по фиксированному id был невозможен.
- Попытка «помочь» дайлу через `PeerCondition::Always` плодила churn в десятки соединений и **ломала регистрацию узлов** — стабильный `Disconnected` дал и меш, и рёбра.

После трёх точечных правок (порт reachability, включённая персистентность ключа, стабильный bootstrap) `placement(min_nodes=2)` вернул валидную раскладку MlxRing, runners поднялись на обоих узлах, и **27B-модель посчиталась распределённо по двум Mac Studio**.

## Что в итоге измерили

| Конфигурация | Скорость (Qwen3.5-27B-4bit) |
|---|---|
| 1 узел через exo | **32 tok/s** |
| 2 узла (pipeline) | **16.5 tok/s** |
| Межузловой линк | 10GbE, 9.4 Gbit/s, RTT 0.6 мс |

Два узла **медленнее** на модели, которая влезает в один — и это правильный результат. Pipeline добавляет межузловой обмен на каждый токен, а выигрыша по памяти нет. **Ценность пула в 1 ТБ — не ускорение, а возможность запустить модель, которая физически не помещается в один Мак.** Ровно поэтому настоящий тест — модель на 671B параметров (DeepSeek-V3.2-8bit, ~756 ГБ на узел), и это тема второй статьи — где обнаружилось, что «собрать кластер» и «посчитать на нём frontier-модель» разделяют ещё четыре слоя проблем.

## Мораль

Семь «очевидных» причин EHOSTUNREACH оказались ложными — настоящей был унаследованный сетевой scope процесса. «Собранный кластер» не считал из-за битого форка движка. «Готовая топология» не давала цикла из-за хардкода порта в reachability-проверке. Общее во всех трёх актах одно: **гипотезу проверяет эксперимент, а не убеждённость**, и дифференциальный тест (то же действие здесь работает, там — нет) указывает на причину быстрее любого рассуждения.
